--- layout: post title: "A lean Lean introduction (DRAFT)" date: 2026-06-18 summary: "Introduction to theorem proving in Lean." tags: "#lean #math" --- The goal in this exposition is to show how theorems are proved in Lean. The primary focus is to construct some intuitive examples that illustrate this. Readers wanting to get just a taste can stop after first order logic and definitionally true theorems. However, the infrastructure developed up to that point does not permit the construction of interesting mathematics. By way of three examples I introduce the three axioms Mathlib uses, soundness of quotients, extensionality for propositions, and the almighty axiom of choice. With these, Lean theorems have been built in diverse topics, from number theory to black holes.

Building blocks

In Lean one can define dependent types. Types, on their own, are things you would see in any programming language, such as String or Float. Let us begin with a few simple examples, starting with a custom Food type, 

inductive Food where
  | banana : Food
  | apple : Food
  | orange : Food
  | sandwich : Food
  | pizza : Food
In regular math-lingo we might express this as "let banana, apple, orange, sandwich, and pizza be Food items". The five entries are the type's constructors, we can use them to construct things which are of type Food, 

def myFavFruit : Food := Food.banana
You can try any snippet in this post in your browser at lean4web, construct yourself an example food and then add a line #check myFavFruit. Hovering your mouse over the #check will display myFavFruit : Food, indicating that your fruit is of type Food. So why do we use the word inductive to define a new type? This is because a constructor, like pizza may take the type itself as an argument, so larger values are built out of smaller ones. However in the case above our constructors took no arguments, as you saw we just defined a food item by writing Food.banana. So in this case, Food is rather uninteresting, it is just a flat list with no self-reference. To illustrate the inductive property let us define a new type, Soup 

inductive Soup where
  | ingredient (f : Food) : Soup
  | recycle (s : Soup) (f : Food) : Soup
In plain words: "a soup is either ingredient f for some food f, or recycle s f built from a pre-existing soup s and one more food f". Because recycle refers back to Soup, we can keep stacking soups out of smaller soups, 

def someFood : Food := Food.pizza
def aTerribleSoup : Soup := Soup.ingredient someFood
def aWorseSoup : Soup := Soup.recycle aTerribleSoup myFavFruit
This we could say is soup(soup(pizza),banana), a pizza-banana soup. Not appetizing, and I am no big fan of soup so we will not pursue the soup type further.
Aside: numbers come for free The same recursive shape gives the natural numbers — a number is either zero or the successor of another, 

inductive Nat where
  | zero : Nat
  | succ (n : Nat) : Nat
So 3 is literally succ 2: a number just is “one more than the one before.” This is refreshingly free of the set-theoretic encoding, where 3 is the set {0, 1, 2} and meaningless questions — like whether {{0}} ∈ 3 — become perfectly well-posed. The machinery that built our lunches is exactly the machinery of real mathematics.
…though type theory smuggles in its own junk Encodings still leave fingerprints. Because Lean represents a rational as a structure with internal fields, the very first entry in junk-theorems-in-lean proves that “the third coordinate of 1/2 is a bijection” — a vacuous truth about an implementation detail, not about the number one-half. Such junk theorems are artifacts of how we build objects; what equality and structure ought to mean, more carefully, is the subject of another post.
Our running example will be a lunch holding two foods, 

inductive Lunch where
  | pack (a b : Food) : Lunch
A def names a value or a function. To define a function we pattern match on how its input was built — here we take a lunch apart and put it back the other way round, 

def swap : Lunch → Lunch
  | Lunch.pack a b => Lunch.pack b a

True by definition

Here is the aha. To prove a statement is to construct a term whose type is that statement — the exact thing def has been doing all along. A proof is a term; a theorem is a definition whose type happens to be a proposition. So we state and prove that swapping a concrete lunch gives what we expect with the very same def we used to build foods and soups, 

def swap_concrete :
    swap (Lunch.pack Food.pizza Food.banana)
  = Lunch.pack Food.banana Food.pizza := rfl
The proof is rfl (reflexivity: everything equals itself), and it needs no axioms — it type-checks because both sides compute to the same lunch. Lean does offer a second keyword, theorem, for exactly this case — a definition whose type is a proposition — but it is the same act of construction. We use it from here on, remembering it is a def in disguise. So too is the fact that swapping any lunch twice returns the original — both sides reduce to the same lunch, 

theorem swap_invol (lb : Lunch) : swap (swap lb) = lb := rfl
We can also prove pure logic this way. A statement lives in Prop (the type of propositions), and the arrow → that typed our functions doubles as “implies”; a proof is still just a term we assemble. A proof of p ∧ q is a pair, written ⟨_, _⟩, with .left and .right to read the two halves back out. Swapping an and is just rebuilding the pair the other way round, 

theorem and_swap (p q : Prop) : p ∧ q → q ∧ p :=
  fun h => ⟨h.right, h.left⟩
A proof of p ∨ q is one side or the other, built with Or.inl (left) or Or.inr (right); to use one we handle both cases with .elim. Swapping an or sends each side to the opposite constructor, 

theorem or_swap (p q : Prop) : p ∨ q → q ∨ p :=
  fun h => h.elim Or.inr Or.inl
With three propositions we can mix the two — here and distributes over or. We split on the ∨, and in each case pair the shared p with whichever side we landed on, 

theorem distrib (p q r : Prop) : p ∧ (q ∨ r) → (p ∧ q) ∨ (p ∧ r) :=
  fun h => h.right.elim
    (fun hq => Or.inl ⟨h.left, hq⟩)
    (fun hr => Or.inr ⟨h.left, hr⟩)
Equality itself holds a small surprise. Any two proofs of the same proposition are equal, and once more the proof is just rfl, 

theorem proofs_agree (p : Prop) (h1 h2 : p) : h1 = h2 := rfl
This is proof irrelevance: to Lean, any two pieces of evidence for a proposition are interchangeable. It fails for ordinary data — two foods need not be equal, so (a b : Food) : a = b := rfl is rejected. Proofs collapse; data does not. That Lean chooses to identify all proofs is a decision of its foundation, and not the only option: there is another, Univalent Foundations, where equality carries structure and two proofs can genuinely differ. That is a story for another post. This already covers a surprising amount. But some things we believe are true cannot be reached by construction alone. There are exactly three, and each is a single axiom.

Quot.sound: related things become equal

A lunch remembers order: pack pizza banana and pack banana pizza are different lunches. But we often want to ignore order. So we declare a relation — "one lunch is the other, swapped", 

inductive SwapRelated : Lunch → Lunch → Prop where
  | swap lb : SwapRelated lb (swap lb)
This relation is symmetric (if x swaps to y then y swaps to x), which we can prove by hand — no rfl, just taking the proof apart and rebuilding it. We can spell the lunch out explicitly, 

theorem swap_symm (x y : Lunch) : SwapRelated x y → SwapRelated y x :=
  fun h =>
    match h with
      | SwapRelated.swap x =>
      match x with
        | Lunch.pack a b => SwapRelated.swap (Lunch.pack b a)
or, since a lunch has a single constructor, let Lean fill in the swapped lunch itself, 

theorem swap_symm' (x y : Lunch) : SwapRelated x y → SwapRelated y x :=
  fun h => match h with
    | SwapRelated.swap x => SwapRelated.swap (swap x)
But it is not reflexive or transitive — a lunch is not swap-related to itself. So SwapRelated is not an equivalence, and on its own it is not equality. To actually treat swapped lunches as the same meal, we form a quotient with the built-in Quot. It comes with two construction operations: one builds the quotient type from a relation, the other places an element into it (as its glued-together class), 

Quot     : (α → α → Prop) → Type            -- the quotient type
Quot.mk  : (r : α → α → Prop) → α → Quot r   -- an element's class
We use them to define our type of order-forgetting meals, 

def Meal : Type := Quot SwapRelated
def Meal.mk (lb : Lunch) : Meal := Quot.mk SwapRelated lb
So Meal.mk lb is the class of lb, the lunch with its order forgotten. Everything so far was ordinary construction. The axiom appears only when we want an actual equality. That axiom is Quot.sound: if two elements are related, their classes are equal, 

Quot.sound : r a b → Quot.mk r a = Quot.mk r b
Feeding it our swap proof gives exactly the equality we wanted — the same two foods, packed either way, are one meal, 

theorem meals_equal (lb : Lunch) : Meal.mk lb = Meal.mk (swap lb) :=
  Quot.sound (SwapRelated.swap lb)

#print axioms meals_equal
-- 'meals_equal' depends on axioms: [Quot.sound]
And now the laws SwapRelated lacked — reflexivity, transitivity — come for free, because Meal uses genuine =, which always has them. That is exactly what the quotient buys: hand it a humble relation, get back equality.

propext: equivalent propositions become equal

Now describe a lunch as a set of foods — a predicate saying, for each food, whether it is included. Membership is then just applying the predicate, and it is a proposition, 

def FoodSet : Type := Food → Prop
def Mem (f : Food) (s : FoodSet) : Prop := s f

def fruitsA : FoodSet := fun f => f = Food.banana ∨ f = Food.apple
def fruitsB : FoodSet := fun f => f = Food.apple  ∨ f = Food.banana
These two sets clearly have the same members. But "same members" means their membership propositions are equivalent, and we want them equal. Constructive logic has no rule turning ↔ into =; that rule is the axiom propext — equivalent propositions are equal, 

propext : (a ↔ b) → a = b
Our two memberships are equivalent because ∨ is symmetric, and propext turns that into an equality, 

theorem mem_same (f : Food) : Mem f fruitsA = Mem f fruitsB :=
  propext ⟨Or.symm, Or.symm⟩   -- the two memberships agree because ∨ is symmetric
To lift this from "equal at each food" to "equal as sets" we need function extensionality, funext — functions that agree at every input are equal (∀ x reads “for every x”), 

funext : (∀ x, f x = g x) → f = g
Applying it to mem_same at every food finishes the job, 

theorem fruits_equal : fruitsA = fruitsB := funext (fun f => mem_same f)
Here is the lovely part: funext is not a fourth axiom — it is a theorem built from Quot.sound. We can prove it ourselves. First quotient the functions by pointwise equality, 

def Pointwise (f g : Food → Prop) : Prop := ∀ x, f x = g x
def QFun : Type := Quot Pointwise
To get a function back out of a quotient we use Quot.lift — the counterpart of Quot.mk, which put things in, 

Quot.lift : (f : α → β) → (∀ a b, r a b → f a = f b) → Quot r → β
We give it a function on representatives plus a proof that the function respects the relation (related inputs give equal outputs), and it descends to the quotient. Here the function is “apply at x”, and the obligation hab x is exactly that pointwise-equal functions agree at x, 

def recover (q : QFun) : Food → Prop :=
  fun x => Quot.lift (fun f => f x) (fun a b (hab : Pointwise a b) => hab x) q
The final piece, congrArg, says applying a function to both sides of an equation preserves it (congrArg f : a = b → f a = f b). So funext is just congrArg recover applied to Quot.sound, 

theorem myfunext {f g : Food → Prop} (h : ∀ x, f x = g x) : f = g :=
  congrArg recover (Quot.sound (r := Pointwise) h)
(The (r := Pointwise) just names which relation we mean, since Lean cannot infer it from h alone.) So set equality quietly leans on two axioms at once, 

#print axioms mem_same      -- [propext]
#print axioms fruits_equal  -- [propext, Quot.sound]   (funext is Quot.sound in disguise)

Classical.choice: existence yields an element

The last axiom is the strongest. Classical.choice takes a mere proof that a type is inhabited and produces an actual element of it, 

Classical.choice : Nonempty α → α
We seldom use it raw. The convenient form is Classical.choose, which pulls a witness out of an existence proof (∃ x, p x), with Classical.choose_spec certifying that the witness really satisfies p, 

Classical.choose      : (∃ x, p x) → α
Classical.choose_spec : (h : ∃ x, p x) → p (Classical.choose h)
A proof of ∃ x, p x is a witness paired with evidence, written ⟨witness, proof⟩. So knowing that a fruit is in our set becomes the power to name one, 

theorem fruitsA_nonempty : ∃ f, fruitsA f := ⟨Food.banana, Or.inl rfl⟩

noncomputable def someFruit : Food := Classical.choose fruitsA_nonempty
theorem someFruit_mem : fruitsA someFruit := Classical.choose_spec fruitsA_nonempty

#print axioms someFruit
-- 'someFruit' depends on axioms: [Classical.choice]
The report still names Classical.choice because choose is built from it. And noncomputable is the axiom confessing its nature: it grants that a fruit exists, but cannot compute which. From this one axiom flows the law of the excluded middle — every proposition is true or false (here ¬p means “p leads to a contradiction”), 

Classical.em : (p : Prop) → p ∨ ¬p
So any property of a lunch is decided one way or the other, 

theorem lunch_decided (P : Lunch → Prop) (lb : Lunch) : P lb ∨ ¬ P lb :=
  Classical.em (P lb)

#print axioms lunch_decided
-- 'lunch_decided' depends on axioms: [propext, Classical.choice, Quot.sound]
Excluded middle is not assumed directly; it is derived from choice, and the derivation (Diaconescu's theorem) draws on all three axioms at once. The three things construction could not reach, together, hand us classical logic.

How much can we do?

With dependent types and these three axioms you can formalize essentially all of working mathematics — analysis, algebra, topology — as Lean's library Mathlib demonstrates daily. But you can never prove everything: Gödel's incompleteness theorems guarantee that any system this expressive has true statements it cannot prove, including its own consistency. The axioms extend our reach into classical mathematics; they do not, and cannot, make the system complete. Some things really are just true.
